# Task Review

Validate a task list (typically `.ralph/fix_plan-F{N}.md`) against quality criteria and its source document (feature spec or PRD) to ensure all requirements are covered, tasks are well-formed, and TDD structure is correct.

## Input

Arguments from command: `$ARGUMENTS`

Expected format: `<fix-plan-path> [spec-or-prd-path]`

- If no arguments: look for `.ralph/fix_plan-*.md` files and try to detect the source document
- If only fix plan provided: search for the source it was generated from

### Source Detection

The task list can be generated from two sources — detect which one:

1. **Feature spec** (preferred pipeline): Look for `.specs/*.md` files. If found, this is the primary reference.
2. **PRD** (legacy pipeline): Look for `PRD-*.md` files. Use when no feature spec exists.

If both exist, ask the user which to validate against.

## Reference Documents

1. The fix plan file (task list to validate)
2. The source document (feature spec or PRD)
3. `CLAUDE.md` from workspace root — project commands and architecture

## Validation Checklist

### 1. Task Quality (per task)

For each task in the fix plan, verify:

- [ ] **Deterministic** — Single unambiguous expected outcome (not vague like "improve" or "refactor")
- [ ] **Verifiable** — Can be validated by running a command (test, lint, type check)
- [ ] **Objective** — No subjective judgment required for pass/fail
- [ ] **Atomic** — Completable in one context window (not too large)
- [ ] **Specific** — Names target file, classes/functions, and expected behavior
- [ ] **Self-contained** — Does not depend on undeclared context

Flag tasks that fail these criteria with specific suggestions for improvement.

### 2. TDD Structure (RED-GREEN pairs)

**This check applies when the task list follows TDD structure (generated by `/feature-to-tasks`).**

For each method in the source document's Internal Interfaces (§ 4), verify:

- [ ] A **RED task** exists (write failing tests) that references the method
- [ ] A **GREEN task** exists (implement the method) that follows the RED task
- [ ] **RED comes BEFORE GREEN** — never the reverse
- [ ] The RED task lists specific test scenarios from the source's § 6
- [ ] The GREEN task references "tests from previous task must pass"
- [ ] No test scenarios from § 6 are missing (every scenario appears in exactly one RED task)
- [ ] No duplicate test scenarios across RED tasks
- [ ] **Inline § traceability**: Every task includes at least one `(§ X.Y)` annotation referencing the spec section it derives from. Scenario codes alone (H1, ER4) are insufficient — they must be accompanied by their source section (e.g., `H1, H2 (§ 6.1)`)

**If the task list does NOT use RED-GREEN pairs** (legacy `/prd-to-tasks` format):
- Report this as a structural finding: "Task list uses implement-then-test pattern. Consider regenerating with `/feature-to-tasks` for TDD structure."
- Still validate the remaining checklist items.

Count and report:

```
TDD Structure
  Methods with RED-GREEN pairs:  X / Y
  Orphan RED tasks (no GREEN):   Z
  Orphan GREEN tasks (no RED):   Z
  RED-after-GREEN violations:    Z
  Test scenarios covered:        X / Y
  Tasks with § traceability:     X / Y
```

### 3. Spec/PRD Coverage

Cross-reference every requirement in the source document against the task list.

#### When source is a Feature Spec:

- **§ 4 Internal Interfaces** — every method signature must have at least one implementation task
- **§ 3 API Contract** — every endpoint must have a wiring task
- **§ 5 Error Scenarios** — every error path must appear in a test scenario
- **§ 6 Test Scenarios** — every scenario (happy path, edge case, error) must appear in a RED task
- **§ 8.3 SQL Migration** — migration tasks must exist if migrations are defined
- **§ 9 Async Messaging** (if present) — event schemas, producer tasks, consumer tasks, and event integration tests must exist

#### When source is a PRD:

- Extract all features from PRD (same process as `/prd-to-tasks`)
- For each feature, find the task(s) that implement it
- For each feature, find the task(s) that test it
- Report features that have no implementation task
- Report features that have no test task

### 4. Phase Structure

Verify the phase ordering makes sense:

- [ ] Foundation phase comes first (models, schema, exceptions)
- [ ] No phase depends on tasks in a later phase
- [ ] In TDD phases: RED tasks precede their GREEN counterparts
- [ ] Structural tasks (models, schema, wiring) are NOT in RED-GREEN pairs (they don't need them)
- [ ] Quality gate tasks present at end of each phase
- [ ] Quality gate commands match the project stack
- [ ] Event tasks (if any) come after core service logic is tested

### 5. Quality Gate Completeness

For the detected stack, verify quality gates include all required checks:

**Python gates:**
1. `ruff check` (lint)
2. `ruff format --check` (format)
3. `pyright` (type check)
4. `pytest --cov` (tests + coverage) — in phases with tests
5. `xenon` (complexity) — in phases with tests
6. `vulture` (dead code) — in phases with tests

**TypeScript gates:**
1. `biome check` (lint + format)
2. `tsc --noEmit` (type check)
3. `vitest run --coverage` (tests + coverage) — in phases with tests
4. `knip` (dead code) — in phases with tests

### 6. Test Coverage Plan

- [ ] Every method from § 4 (Internal Interfaces) has at least one test scenario
- [ ] All happy path scenarios from § 6.1 are covered
- [ ] All edge case scenarios from § 6.2 are covered
- [ ] All error case scenarios from § 6.3 are covered
- [ ] Integration tests exist for API endpoints (REST + MCP)
- [ ] Event integration tests exist (if § 9 is present): producer emits → consumer receives → side effect verified

### 7. Event Coverage (only if § 9 exists in source)

If the source spec has an Async Messaging section, verify:

- [ ] Event schema tasks exist for each event in § 9.3
- [ ] Producer RED-GREEN pairs exist for each event in § 9.1
- [ ] Consumer RED-GREEN pairs exist for each event in § 9.2
- [ ] Consumer tests verify idempotency (processing same event twice)
- [ ] Channel/topic configuration task exists if § 9.5 defines channels

### 8. Task Sizing

Flag tasks that appear too large or too small:

- **Too large**: Tasks that mention 3+ classes/functions to implement, or span multiple modules
- **Too small**: Tasks that are trivial one-liners (e.g., "create empty __init__.py")
- Suggest splitting large tasks or merging trivial ones

## Output Format

```
Task Review -- <fix plan filename>
Source: <spec or PRD filename>
Pipeline: feature-spec | legacy (prd-to-tasks)
========================================

  1.  Task Quality:           X / Y well-formed
  2.  TDD Structure:          PASS / ISSUES / N/A (legacy)
  3.  Spec/PRD Coverage:      X / Y requirements covered (Z%)
  4.  Phase Structure:        PASS / ISSUES FOUND
  5.  Quality Gates:          PASS / INCOMPLETE
  6.  Test Coverage:          PASS / GAPS FOUND
  7.  Event Coverage:         PASS / GAPS / N/A (no events)
  8.  Task Sizing:            PASS / ADJUSTMENTS NEEDED
  ----------------------------------------
  Overall:                    PASS / NEEDS REVISION
```

### TDD Pairing Detail

```

-------------------------------|----------------|----------------|----------

set_scopes_batch()             | Phase 2, #3    | Phase 2, #4    | 3/3
list_scopes()                  | Phase 2, #5    | Phase 2, #6    | 3/3
<method_name>                  | MISSING         | Phase 3, #1    | 0/2
```

### Spec/PRD Coverage Detail

```
Requirement                            | Task(s)             | Status
---------------------------------------|---------------------|--------
§ 4: set_node_scope()                  | Phase 2, #1-#2      | COVERED
§ 5, row 3: batch bad node rollback    | Phase 2, #3 (RED)   | COVERED
§ 6.2, #4: batch duplicate node_ids    | MISSING              | GAP
§ 9.1: scope.assigned event            | Phase 4, #2-#3      | COVERED
```

### Detailed Findings

For each issue found, provide:

```
[CATEGORY] Task "<task text snippet>"
  Issue: <what's wrong>
  Suggestion: <how to fix>
```

Categories: `QUALITY`, `TDD`, `COVERAGE`, `PHASE`, `GATE`, `EVENTS`, `SIZING`

## Tone

Be precise and actionable. The goal is to catch gaps before implementation begins, saving time and context budget. Every finding should include a concrete fix suggestion.
